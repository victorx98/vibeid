# 新 ATS 評分系統測試指南

## 快速開始

### 1. 啟動開發伺服器
```bash
npm run dev
# 應該在 http://localhost:3000 啟動
```

### 2. 測試流程
1. 打開 http://localhost:3000
2. 上傳一份簡歷（PDF/DOC/DOCX）
3. 輸入目標職位（如 "Software Engineer"）
4. **可選**：輸入職位描述（JD）
5. 點擊「分析簡歷」
6. 等待分析完成並跳轉到 `/sales` 頁面

### 3. 驗證新系統

#### 在 `/sales` 頁面檢查：

**第一步：查看 ATS 得分卡片**
- [ ] 顯示 0-100 的分數圈
- [ ] 狀態標籤正確（「通過」/「勉強通過」/「未達標」/「不通過」）
- [ ] **新增**：顯示風險等級（低/中/高）- 彩色徽章
- [ ] **新增**：顯示評分上下文（"提供 JD" 或 "通用方向估算"）

**第二步：點擊「查看評分詳情」展開**
- [ ] 顯示 5 個維度的分數條形圖：
  - A. 解析與格式兼容性 (0-20)
  - B. 信息完整性與結構組織 (0-20)
  - C. 內容質量與成果表達 (0-35) ← 最重要，分數最高
  - D. 崗位關鍵詞與匹配性 (0-15)
  - E. 最終投遞完成度 (0-10)
- [ ] 分數總和應等於主分數

**第三步：查看問題和改進建議**
- [ ] 「關鍵問題」部分顯示 4 項主要問題
  - 每項包含：嚴重度徽章（高/中/低）+ 問題描述 + 影響說明
- [ ] **新增**：「優先改進建議」顯示 3 項具體行動
  - 每項包含：序號 + 具體行動文本 + 預期效果
- [ ] **新增**：「提分預期」提示框（藍色背景）
  - 格式如："補全量化信息後預計提升 4-7 分"
- [ ] **新增**：「簡歷亮點」部分顯示 0-N 個優勢標籤

#### 其他頁面檢查：

**在 `/result` 頁面：**
- [ ] 「格式合規檢測」警示正常工作
- [ ] 使用新的 A 維度分數（不是舊的 format_compliance）
- [ ] 分數轉換公式正確：(A_format_parsing / 20) * 100

**在 `/upsale` 頁面：**
- [ ] 「ATS 優化詳情」組件顯示原始評分
- [ ] 使用 `ats_score` 欄位（新格式）或 `final_score`（舊格式）
- [ ] 優化預估計算：原始分 + 25 分（上限 95）

## 測試場景

### 場景 1：有 JD 的高分簡歷（測試新系統的上限）
**目標**：驗證新系統是否能正確識別優質簡歷

**簡歷特徵**：
- 清晰的單欄格式
- 具有量化成果的項目經歷（如「提升 30% 效率」）
- 完整的技能部分
- 專業的格式和拼寫
- 包含目標崗位相關的關鍵詞

**預期結果**：
- C 維度分數應為 30-35（最重要）
- 總分應 > 75
- 風險等級應為「低風險」
- 關鍵問題少於 2 項

---

### 場景 2：無 JD 的中等簡歷（測試保守評分）
**目標**：驗證無 JD 時的評分模式

**簡歷特徵**：
- 基本結構完整
- 部分成果缺乏量化
- 普通格式

**預期結果**：
- D 維度分數 ≤ 13（因為無 JD）
- 評分上下文應顯示「通用方向估算」
- 系統應專注於通用的格式和結構評估

---

### 場景 3：低分簡歷（測試向下識別能力）
**目標**：驗證新系統是否能準確識別問題簡歷

**簡歷特徵**：
- 多列布局或表格
- 大量空泛詞彙（"responsible for", "involved in"）
- 缺乏量化
- 多處拼寫錯誤

**預期結果**：
- C 維度分數應低於 15
- 關鍵問題應明確指出具體缺陷
- 優先改進建議應有可操作性
- 風險等級應為「高風險」
- 提分預期應給出明確的改進路徑

## 常見問題排查

### 問題：看不到新的 5 維度評分
**解決**：
1. 硬刷新瀏覽器（Ctrl+Shift+R 或 Cmd+Shift+R）
2. 清除 localStorage：在開發者工具中執行 `localStorage.clear()`
3. 檢查 Network 標籤確認 API 返回新格式

### 問題：舊簡歷分析結果無法顯示
**解決**：
- 新系統包含向後相容邏輯
- 檢查瀏覽器控制台是否有錯誤
- 點擊「查看評分詳情」時，應顯示「LEGACY 4-Dimension bars」

### 問題：優先改進建議為空
**解決**：
1. 確認 API 返回的 `priority_improvements` 字段
2. 檢查是否需要等待 Claude API 完全生成
3. 查看後端 logs
